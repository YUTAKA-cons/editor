# 🚀 Java 最適化 & 規模最小化ガイドライン (Ultra-Lean Edition)

## 1. データ構造とメモリ効率
- **Setの原則**: 重複を許さない、または順序が不要なコレクションは、必ず `HashSet` または `TreeSet` を使用せよ。`List` を回して `contains` でチェックする実装は「重罪」とみなす。
- **プリミティブ優先**: `Integer`, `Double` 等のラッパークラスは、`null` が必須な場合（DBのNull許容カラム等）を除き禁止。必ず `int`, `double` を使用し、オートボクシングによるメモリ消費を回避せよ。
- **初期容量の指定**: `Map` や `List` の要素数が事前に予測できる場合は、`new ArrayList<>(100)` のように初期容量を指定し、内部配列の再確保（Resize）コストを削減せよ。

## 2. インスタンス制御と再利用
- **Singletonの徹底**: 状態を管理するManagerクラスや、共通の設定クラスは必ず `Singleton` パターンで実装し、`new` の乱立を防げ。
- **Staticの活用**: 状態を持たないロジック（計算、変換等）は `static` メソッドとして集約せよ。不要なインスタンス化を伴う「インスタンスメソッド」は排除せよ。
- **文字列結合**: ループ内での `+` 結合は厳禁。必ず `StringBuilder` を使用せよ。ただし、単発の結合であれば可読性を優先し `String.join` や `String.format` を検討せよ。

## 3. 計算量とロジックの簡素化
- **Stream APIの適材適所**: 10要素以下の単純なフィルタリングに `stream()` を使うのはオーバーヘッドが大きいため、従来の `for-each` を優先せよ。逆に複雑な加工は Stream を使い、中間変数を最小化せよ。
- **遅延評価**: 重い処理や大きなオブジェクトの生成は、実際に必要になるまで遅延（Lazy Initialization）させよ。
- **キャッシュ戦略**: 同一入力に対して同一結果を返す重い計算は、計算結果をメンバ変数にキャッシュ（メモ化）し、二度実行するな。

## 4. 規模最小化（Anti-Bloat）
- **既存資産の再利用**: 新規メソッドを作る前に、既存の `utils` クラスに同様の機能がないか必ずスキャンせよ。
- **SRP（単一責任原則）の遵守**: 1つのクラスに機能を詰め込みすぎるな（God Classの禁止）。ただし、分割しすぎてクラス数が増えすぎる場合は、内部クラス（Inner Class）としてカプセル化し、プロジェクトのファイル肥大化を抑制せよ。
- **AI親和性の確保**: メソッドは20行以内を目指せ。AIが一度に理解できるコンテキストを最適化することで、バグ混入と冗長なコード生成を防ぐ。
